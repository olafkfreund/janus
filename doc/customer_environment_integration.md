# Customer Technical Environment Integration Analysis

This document outlines how the **MCP API Gateway & Portal** (codenamed "Janus") integrates, connects, and operates within the Customer-built technical environment:
*   **Hosting**: AWS
*   **Runtime**: Amazon EKS (API + MCP server, Concorde Data Platform), container images in Amazon ECR, LCH Group Portal integration.
*   **Data/AI Services**: AWS S3, Concorde data platform (Snowflake).
*   **DevOps**: DX1, Docker, GitLab CI/CD.
*   **Integrations/Telemetry**: LCH Group Portal, Concorde, DataDog, Langfuse.

---

## 1. Hosting & Runtime Integration (AWS, EKS, ECR)

### Containerization (ECR & Docker)
The gateway repository contains a multi-stage, production-ready `Dockerfile` that compiles the Go binary statically and copies it into a secure, minimal Alpine runner.
*   **Security Compliance**: The container runs as a non-root user (`gateway` group and user, UID/GID 10001) and drops all system capabilities, satisfying Customer security baselines.
*   **ECR Push**: During the build step, the GitLab CI/CD pipeline compiles, tags, and pushes the image directly to **Amazon ECR**.

### Orchestration (Amazon EKS)
The Kubernetes deployment configuration (`k8s-deployment.yaml`) is directly compatible with Amazon EKS:
1.  **Stateless Scale-Out**: To run in a high-availability EKS cluster, the gateway database must be transitioned from SQLite (`DATABASE_PATH`) to a shared **Amazon RDS (PostgreSQL/Aurora)** instance. We achieve this by configuring the `DATABASE_URL` environment variable.
2.  **SSE Sticky Sessions**: EKS pods are fronted by NGINX Ingress or AWS Application Load Balancer (ALB). The ingress contains annotations for cookie-based session affinity (`nginx.ingress.kubernetes.io/affinity: "cookie"`). This guarantees that Server-Sent Events (SSE) connections and subsequent JSON-RPC tool POST calls from LLM clients map to the same pod replica.
3.  **Liveness/Readiness Probes**: The EKS deployment config implements HTTP health checks on `/` to handle self-healing and rolling updates automatically.

```
                  [ AWS EKS Cluster ]
                   
                      ALB/Ingress 
                     (Sticky Cookie)
                           │
             ┌─────────────┼─────────────┐
             ▼             ▼             ▼
          [Pod 1]       [Pod 2]       [Pod 3]  <── (mTLS / Token Auth)
             │             │             │
             └─────────────┼─────────────┘
                           ▼
                 [ Amazon RDS Postgres ]
```

### AWS Security Integration (IRSA & IAM)
Instead of embedding static API keys or cloud tokens inside deployment specs, EKS leverages **IAM Roles for Service Accounts (IRSA)**:
*   By setting `VAULT_PROVIDER=aws` in the deployment environment, the gateway uses the official AWS SDK for Go to authenticate.
*   It assumes the pod's OIDC-backed IAM role to fetch dynamic downstream API credentials directly from **AWS Secrets Manager**, decrypted using **AWS KMS** keys.

> **Encrypted-at-rest even without a cloud vault.** Where the `local` or `postgres` vault providers are
> used (e.g. air-gapped or single-node tiers), secrets are sealed with AES-256-GCM at rest — the `local`
> vault file is encrypted with `VAULT_ENCRYPTION_KEY` (falling back to `JWT_SECRET`) and any pre-existing
> plaintext file is transparently migrated on first load.

### Client Authentication: Janus Tokens or Enterprise OAuth 2.1
LLM clients authenticate to the MCP endpoints in one of two ways, and both can run side by side:
*   **Janus client tokens** — hashed-at-rest bearer tokens issued in the portal/CLI, scoped by tool-name
    globs and role. Ideal for service agents and CI.
*   **OAuth 2.1 resource server** — set `OAUTH_ENABLED=true` with `OAUTH_RESOURCE_URI` and
    `OAUTH_AUTHORIZATION_SERVERS`, and end-user MCP clients already federated with the Customer IdP (Okta /
    Entra ID) can present their **audience-bound** access tokens directly. The gateway publishes RFC 9728
    protected-resource metadata, issues `WWW-Authenticate` challenges, and validates the token's `aud`
    against this gateway (RFC 8707), failing closed on any error. This lets Janus slot into an existing
    enterprise SSO estate without minting a second class of credential.

## 2. Data & AI Services Integration (S3 & Snowflake)

### Concorde Data Platform (Snowflake)
*   **Architectural Principle (Critical)**: To maintain governance, the MCP Facade **never** directly accesses Snowflake to run ad-hoc SQL queries generated by LLMs.
*   **Integration Route**: Customer's Concorde platform engineers expose Snowflake data via governed REST API microservices (e.g., hosted in EKS or API Gateway). 
*   The MCP Gateway maps these REST endpoints as tools. The LLM translates user requests into structured API calls, and the gateway queries the Snowflake REST wrappers, enforcing schema validation, rate-limiting, and auditable metrics.
*   **Onboarding via OpenAPI import**: because the Concorde/Snowflake REST wrappers are described by an OpenAPI 3.x spec, they do **not** have to be mapped tool-by-tool. Point the gateway at the spec and it generates the connection plus one MCP tool per operation in one step:

    ```bash
    # Preview first (writes nothing), then apply with a namespacing prefix
    mcp-cli import openapi https://concorde-api.eks.internal/openapi.json --dry-run --prefix concorde_
    mcp-cli import openapi https://concorde-api.eks.internal/openapi.json --prefix concorde_
    ```

    The import sets the connection's `auth_type` from the spec's security schemes; the actual bearer token is still registered separately in the vault (credentials are never imported). This keeps Concorde onboarding a repeatable, reviewable step in the platform team's workflow.

### AWS S3
*   For reading historical documents (PDFs, CSVs) or saving generated AI collateral reports, the gateway queries downstream REST APIs which interface with AWS S3 using the AWS SDK.
*   To prevent exposing raw S3 buckets, the REST APIs generate **AWS S3 pre-signed URLs** with a short expiration window (e.g., 15 minutes). The MCP server routes these URLs back to the LLM client, allowing users to securely download files.

## 3. DevOps & CI/CD Pipeline (DX1, Docker, GitLab)

The repository's structure is optimized for **GitLab CI/CD** pipelines. The integration flow involves:

```
[ Git Commit ] ──> [ GitLab CI/CD Pipeline ] ──> [ Amazon ECR ] ──> [ Amazon EKS ]
                        │
                        ├─> Lints & Syntax Check (`just validate`)
                        ├─> Unit & Integration Tests (`go test`)
                        └─> Docker Build & Security Scan (Trivy)
```

1.  **Lints & Tests**: The pipeline runs `just validate` which executes `go vet` and tests.
2.  **Security Scans**: The Docker build step incorporates scanning utilities (e.g., Trivy or GitLab container scanners) to inspect the image layers before ECR upload.
3.  **Deployment**: On a successful build, GitLab pushes the image to Amazon ECR and updates the EKS cluster (using `kubectl` or Customer GitOps engines like ArgoCD).

## 4. Integrations & Telemetry (DataDog & Langfuse)

### DataDog Monitoring
DataDog integrates natively with the Go gateway:
1.  **Prometheus Scraping**: The gateway exposes standard metrics on `/metrics`. The DataDog Agent running in the EKS cluster is configured via pod annotations to scrape these metrics, feeding data into dashboards for request rates, execution duration, and tool errors.
2.  **Structured JSON Logging**: The gateway outputs structured JSON logs to standard output. DataDog log collection catches these and indexes fields (`caller_id`, `tool_name`, `duration_ms`, `status`) automatically. With `REDACTION_ENABLED=true`, PII/secrets in tool arguments and downstream responses are masked (`[REDACTED:<class>]`) *before* they can reach these logs — the audit trail records only per-class hit counts, so DataDog never indexes a raw email, card, or key.
3.  **APM Tracing**: The Datadog tracer (`dd-trace-go`) can be imported into the gateway main logic, propagating trace headers across the EKS pods down to the REST microservices.

### Langfuse (LLM Observability)
Langfuse tracks prompts, latency, traces, and token costs:
*   **Client-Side Tracing (Best Practice)**: The client application (e.g., Concorde UI or LCH Portal Integration) initializes the Langfuse SDK. When the client calls the LLM, the entire trace (prompt -> tool call to the MCP gateway -> tool response -> final summary) is logged directly to Langfuse.
*   **Gateway-Side Tracing**: The MCP Gateway can be configured to emit custom trace payloads to the Langfuse API endpoint (`/api/public/ingestion`) on every tool execution, tracking duration and schema sizes.
